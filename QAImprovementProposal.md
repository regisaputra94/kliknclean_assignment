# QA Improvement Plan — First 30/60/90 Days

## First 30 Days — Understand, Stabilize, Build Trust

As the first QA engineer, my initial focus would not be maximizing test coverage immediately, but understanding the product, release process, and existing quality risks.

### Key Activities

* Learn the current development and release workflow.
* Analyze recent production incidents and identify recurring risk areas.
* Define critical user journeys and establish a baseline smoke test suite.
* Integrate smoke tests into CI/CD to provide immediate feedback on every pull request.

### Deliverables

* Risk-based testing strategy.
* Automated smoke test suite running in CI.
* Prioritized quality improvement backlog.

---

## First 60 Days — Expand Coverage and Shift Left

After establishing a stable baseline, I would focus on increasing confidence through scalable testing practices.

### Key Activities

* Expand automation coverage from smoke tests to regression testing.
* Prioritize API-level automation following the test pyramid approach.
* Introduce a lightweight defect management and triage process.
* Participate earlier in requirements discussions to improve testability and acceptance criteria.
* Introduce API contract testing for critical integrations.

### Deliverables

* Regression suite covering major business risks.
* Standardized defect workflow and ownership model.
* Shift-left practices integrated into the team's development process.

---

## First 90 Days — Measure, Scale, and Drive Quality Culture

Once foundations are in place, the focus shifts to measurement and continuous improvement.

### Key Activities

* Define and track quality metrics such as escaped defects, automation stability, and defect resolution trends.
* Strengthen collaboration between QA and engineering through shared ownership of quality.
* Review incident learnings and continuously improve test coverage based on real production risks.
* Use collected data to identify future quality investments.

### Deliverables

* Quality dashboard with measurable KPIs.
* Adopted Definition of Done including testing expectations.
* Data-driven proposal for future QA initiatives and tooling investments.

---

## Guiding Principle

The goal is not achieving 100% automation coverage, but building reliable, maintainable, risk-based quality practices that provide real business value and can scale with the engineering team.
